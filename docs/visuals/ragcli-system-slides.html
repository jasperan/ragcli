01 / 07 project briefing

ragcli, from Oracle vectors to a usable product shell.

This repo packages the same retrieval system as a terminal app, a FastAPI service, and a React frontend. Under the hood it keeps documents, chunks, graph relationships, traces, evals, sync events, and sessions in Oracle 26ai while Ollama stays responsible for local embeddings and answer generation.

04 / 07 runtime query path

Queries move through retrieval, memory, and logging before the answer is done.

where hybrid search fits

Vector is the baseline, hybrid is the upgrade path.

The repo already has a HybridSearch class that fuses vector, BM25, and graph-driven chunk IDs via reciprocal rank fusion. That means the system is wired for stronger retrieval than a pure embedding nearest-neighbor pass, even when the current query path stays simple.

vector similarity BM25 fallback graph hop retrieval session memory
why it matters

It remembers what a one-shot RAG bot forgets.

Sessions, logs, eval rows, and feedback all make later tuning possible. The repo isn’t locked into a “hit enter, get text, lose context” lifecycle.

The big takeaway: this repo already spans ingestion, retrieval, memory, graph enrichment, evaluation, sync, API delivery, and UI inspection. The whole thing is opinionated around Oracle as the storage brain and Ollama as the local model runtime. That makes ragcli more interesting than a toy chatbot wrapper, it’s a compact retrieval platform with multiple ways to use it.
