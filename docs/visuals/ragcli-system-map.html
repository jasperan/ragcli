01 — overview
Project posture

One repo, 3 surfaces, 1 Oracle-backed retrieval engine.

ragcli exposes the same core system through a menu-driven CLI, a REST API, and a React frontend. Documents get chunked, embedded, and written into Oracle vector tables. Queries come back through retrieval, local generation, and optional context layers like sessions, graph search, evaluation, and sync automation.

CLI: ragcli.py API: ragcli/api/server.py UI: frontend/src/pages Core: ragcli/core/rag_engine.py
System center

Oracle + Ollama

Oracle stores documents, chunks, graph data, sessions, eval results, sync events, and feedback. Ollama handles local embedding and chat inference. That combo keeps the stack local-first without giving up persistent retrieval state.

02 — runtime flows
Ingest path

Upload to searchable state

The ingest path is more than a file save. It validates format, extracts text, chunks content, generates embeddings, inserts Oracle rows, and optionally extracts graph entities and relationships from each chunk.

step 01 Preprocess OCR-aware text extraction in document_processor.py.
step 02 Chunk Split text into retrieval-sized units with token metadata.
step 03 Embed Generate vectors through Ollama or Oracle embeddings.
step 04 Persist Write documents and chunks into Oracle vector tables.
step 05 Extract graph Populate KG_ENTITIES, KG_RELATIONSHIPS, and chunk links.
Query path

Prompt to grounded response

The query path adds optional session memory before retrieval, then retrieves chunks, assembles context, generates the answer, logs the interaction, and stores the turn if a session is active.

step 01 Rewrite When session_id exists, recent turns and summaries can rewrite follow-up questions.
step 02 Retrieve search_chunks() pulls the most relevant chunk set and metrics.
step 03 Generate Ollama receives context + user query and returns the grounded answer.
step 04 Log Query rows, results, timings, and optional embeddings are persisted for later analysis.
step 05 Remember Session turns and rolling summaries keep later prompts contextual.

03 — intelligence layers
Search

Vector, BM25, graph

HybridSearch fuses vector similarity, BM25, and graph-derived chunk IDs with reciprocal rank fusion. That keeps short literal matches from getting buried behind embeddings.

Memory

Session turns + summaries

SessionManager, QueryRewriter, and ContextManager hold recent turns, rewrite follow-ups, and decide when the session needs a rolling summary.

Agents

4-role orchestration

The agent layer is built around Planner, Researcher, Reasoner, and Synthesizer roles, plus a trace object that can preserve the reasoning path for inspection.

Knowledge graph

Chunk-linked entities

The graph store lets the repo keep entities and relationships alongside the chunks they came from, which is exactly the glue hybrid retrieval usually lacks.

Feedback + eval

Score the answers later

Feedback collectors, Wilson score chunk quality, synthetic evaluation runs, and report generation turn the repo into something you can tune over time.

Sync

Keep docs live

SyncScheduler and watcher components can monitor paths, repos, or feeds, then diff and re-ingest only the pieces that changed.

Why this matters: a lot of RAG repos stop at upload + query. ragcli bolts on the layers you need after the first demo: traces, memory, graph enrichment, automated evaluation, and refresh pipelines.

How it ships

Local, packaged, or containerized

ragcli is set up to run from source, ship through PyPI, start as an API service, or boot as a Docker Compose stack with Oracle and Ollama nearby. That makes the repo practical for tinkering and believable as a real deployment baseline.

  • pip install oracle-ragcli for the packaged path
  • python ragcli.py api --port 8000 for service mode
  • cd frontend && npm run dev for the React shell
  • docker-compose up -d for the full local stack

Bottom line: the repo is structured like a toolkit and a product at the same time. You can run it in a terminal, wire it into another app over HTTP, or hand it to a user who just wants a search box and a graph view.
